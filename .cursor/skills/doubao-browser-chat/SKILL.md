# 豆包浏览器对话测试

在浏览器中操作豆包页面完成一轮或多轮对话，适用于端到端验证、提示词效果比对、回答质量抽检。

## 适用触发词

- 豆包
- doubao
- `https://www.doubao.com/chat/`
- 浏览器操作豆包
- 豆包对话测试

## 执行原则

1. 优先使用 `browser-use` 子代理执行连续网页操作，避免把复杂步骤拆得过碎。
2. 每次测试都要先确认是否已登录；如需人工扫码/授权，暂停并提示用户完成。
3. 发送消息后等待回答稳定（不再继续追加）再采集最终结果。
4. 只记录必要信息：测试问题、模型回答、页面异常、是否通过。
5. 优先复用用户已附加的豆包标签页（若提供了 Browser ID）。

## 能力自检（必须先做）

在正式执行用户问题前，先做一轮最小可用性验证：

1. 打开或切换到豆包页面。
2. 发送 `你好`。
3. 等待并读取豆包回复。
4. 若该步骤失败，立即返回“阻塞”，不要继续后续测试。

自检通过标准：

- 成功定位输入框并发送消息。
- 能读取到至少一段非空回复文本。
- 页面无明显崩溃或死循环加载。

## 标准流程

### 1) 启动浏览器任务

当用户要求“做豆包浏览器操作/对话测试”时，启动 `browser-use` 子代理并给出明确目标：

- 优先复用用户附加标签页（如 `Browser ID: xxx`）
- 打开 `https://www.doubao.com/chat/?from_login=1`
- 检查页面是否可交互
- 先执行“发送你好”能力自检
- 确认登录状态
- 在输入框发送指定问题
- 等待并读取回答
- 返回结构化测试结果

### 2) 登录态处理

- 若已登录：直接进入发送消息步骤。
- 若未登录：明确提示用户先在浏览器完成登录，然后继续后续步骤。
- 若出现风控/验证码：记录为“阻塞项”，不要伪造测试结论。

### 3) 对话执行

- 在同一会话按用户给定问题发送。
- 若用户未给问题，使用默认探测问题：
  - `请用三句话介绍你自己，并说明你能帮我做什么。`
- 等待回答渲染完成后提取文本。
- 如回答过长，提取前 300-500 字并标注“已截断”。

### 4) 结果回传格式

返回时统一使用以下结构：

```markdown
豆包对话测试结果
- Browser ID: <有则填写，无则写 N/A>
- 访问地址: https://www.doubao.com/chat/?from_login=1
- 能力自检: 通过 / 未通过
- 自检回复: <发送“你好”后收到的回复或失败原因>
- 登录状态: 已登录 / 未登录 / 需人工验证
- 测试问题: <用户问题或默认问题>
- 回答摘要: <提取到的回答>
- 页面异常: 无 / <异常描述>
- 结论: 通过 / 未通过 / 阻塞
```

## 推荐子代理提示词模板

在调用 `browser-use` 子代理时，可直接复用下面模板并替换变量：

```text
请在浏览器中执行豆包对话测试：
1) 优先切换到用户提供的豆包标签页（若有 Browser ID），否则打开 https://www.doubao.com/chat/?from_login=1
2) 判断当前是否已登录；若未登录，停在登录页并返回“需人工登录”
3) 先执行能力自检：发送“你好”，并提取回复
4) 若自检失败，直接返回阻塞，不继续
5) 若自检通过，在聊天输入框发送：<QUESTION>
6) 等待回答完成，提取主要回答内容
7) 返回：
   - Browser ID（若可见）
   - 能力自检是否通过
   - “你好”回复摘要
   - 登录状态
   - 发送的问题
   - 回答摘要
   - 页面是否报错或卡住
   - 最终结论（通过/未通过/阻塞）
```

## 失败与重试策略

- 页面加载失败：刷新 1 次后重试。
- 输入框不可用：重新打开聊天页再试 1 次。
- 若子代理环境无浏览器操作能力：立即标记阻塞，并要求用户开启 browser-use 能力或改为人工执行步骤。
- 连续两次失败：标记为“阻塞”，返回失败原因与建议（网络、登录、风控）。

## 注意事项

- 不在结果中暴露用户隐私信息（手机号、邮箱、验证码）。
- 不编造未实际看到的页面状态或回答内容。
- 仅在用户明确要求时执行多轮对话压测。
