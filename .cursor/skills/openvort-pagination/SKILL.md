# 分页参数一致性规范

## 核心原则

**前端请求的 `page_size` 不得超过后端接口的 `le` 上限。** 每次新增或修改涉及分页的代码时，必须同时检查前后端的一致性。

## 分页上限分级

根据使用场景，`page_size` 上限分为两档：

| 场景 | 后端 `le` 值 | 前端建议值 | 说明 |
|------|-------------|-----------|------|
| 分页列表（表格/卡片） | `100` | `20` ~ `100` | 用户逐页浏览，无需一次全量加载 |
| 下拉选择器/全量加载 | `200` | `100` ~ `200` | 需要一次加载全部选项的场景 |

## 后端规则

### 规则 1：新增列表接口必须声明 `page_size` 上限

所有接受 `page_size` 参数的接口必须使用 `Query` 约束，禁止无上限：

```python
# 分页列表场景
page_size: int = Query(20, ge=1, le=100)

# 下拉选择器/全量加载场景
page_size: int = Query(20, ge=1, le=200)
```

### 规则 2：选择正确的上限档位

判断标准：

- 接口主要被表格/卡片列表调用 -> `le=100`
- 接口会被下拉选择器、批量编辑器等需要全量加载的组件调用 -> `le=200`
- 同一接口同时服务两种场景 -> `le=200`（取较大值）

### 规则 3：禁止不统一的零散值

不要出现 `le=50`、`le=150`、`le=300` 等非标准值。统一使用 `100` 或 `200`。

## 前端规则

### 规则 1：下拉选择器/全量加载统一使用 `page_size: 200`

当前端需要一次性加载全部数据用于下拉选择（迭代、版本、需求、仓库等），统一使用 `page_size: 200`：

```typescript
const res = await getVortflowIterations({ project_id: pid, page_size: 200 });
```

### 规则 2：分页表格默认 `page_size: 20`

表格分页场景使用默认值，不要随意放大：

```typescript
const res = await getVortflowStories({ page: 1, page_size: 20 });
```

### 规则 3：修改前端 `page_size` 时必须检查后端上限

如果要将前端的 `page_size` 从 `100` 改为 `200`，必须先确认后端对应接口的 `le` 值是否 >= `200`。不确认就提交是错误的。

## 执行检查

每次新增或修改分页相关代码时：

1. **新增后端列表接口** -> 根据场景选择 `le=100` 或 `le=200`，禁止遗漏 `le` 约束。
2. **前端调用列表接口** -> 确认 `page_size` 值不超过后端 `le` 上限。
3. **前端将 `page_size` 提升到 `200`** -> 必须同步检查/调整后端 `le` 值。
4. **后端将 `le` 从 `200` 降到 `100`** -> 必须全局搜索前端调用处，确认无超限请求。

## 反例

### 反例 1：前端请求超过后端上限

```typescript
// 后端: page_size: int = Query(20, ge=1, le=100)
// 前端:
const res = await getVortflowVersions({ page_size: 200 }); // 400 错误
```

必须先将后端改为 `le=200`，或将前端改为 `page_size: 100`。

### 反例 2：后端接口不声明上限

```python
# 没有 le 约束，可以请求任意大的 page_size
page_size: int = Query(20, ge=1)
```

必须加上 `le=100` 或 `le=200`。

### 反例 3：使用非标准上限值

```python
page_size: int = Query(20, ge=1, le=50)   # 非标准
page_size: int = Query(20, ge=1, le=150)  # 非标准
```

统一使用 `100` 或 `200`。
