# 代码审查

当用户提到代码审查、Code Review、CR 相关话题时，按以下规范指导。

## 审查要点

1. **功能正确性** — 代码是否实现了需求描述的功能
2. **边界处理** — 空值、异常、并发等边界场景是否覆盖
3. **安全性** — SQL 注入、XSS、敏感信息泄露等
4. **可读性** — 命名是否清晰、逻辑是否易懂、注释是否充分
5. **性能** — 是否有明显的性能问题（N+1 查询、大循环等）

## 审查反馈格式

```
### CR 反馈 — [模块/文件名]

**必须修改：**
- [ ] 问题描述 + 建议修改方式

**建议优化：**
- [ ] 优化建议

**亮点：**
- 值得肯定的设计或实现
```

## 注意事项

- 反馈语气友好专业，指出问题的同时给出具体建议
- 区分"必须修改"和"建议优化"的优先级
- 如果用户提供了代码片段，直接针对代码给出审查意见
